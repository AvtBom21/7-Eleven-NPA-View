Core Business Flow
Stages are ordered:
1. Category
2. Manager
3. Director
4. Present List
5. Final Information
6. Setup Product ID

Important distinction:
- storedSheet: the sheet that stores the current process/source row state.
- effectiveSheet: the stage where the product is considered to be now, by business rules.
- uiPlacement: whether chat appears in Edit Workspace or Process.
- storageSource: whether SSV Note Product is physically read/written in Category or Database.

Business Flow State Machine
Central backend helper is in Code.js:
- _normalizeBusinessFlowStatus_(value)
- _resolveBusinessFlowState_(storedSheet, status, options)
- _buildBusinessFlowStageTimeline_(state)

Rules:
- Processing / process / in progress / in process / progressing means forward to next stage.
- Rejected / Reject / any status containing reject means rejected at the stored sheet, do not move forward.
- Pending does not move forward.
- Done / Complete / Completed does not move forward under the current confirmed rules.
- Last stage plus Processing does not go past Setup Product ID.
- Unknown stage/status should fail safe without crashing; state includes warnings.

Process timeline and labels must use this state machine. Frontend must not reimplement forward/reject rules independently.

Access Model
Access is token-based via User_Master and server bootstrap.
Important access fields:
- selectedView: CAT, MAN, DIR.
- selectedSheetName: Category, Manager, Director, etc.
- allowedViews.
- currentViewCanEdit.
- scope / categoryScope.

Read/write rules:
- Read product data/chat only if current access/scope allows it.
- Write product data only if current view can edit.
- Chat add requires current view can edit.
- Chat edit/delete requires current view can edit and actor email must match message PIC exactly.

SSV Note Product / Internal Chat
SSV Note Product is no longer a normal editable product text field. It is a chat transcript.

Chat storage format per message line:
PIC|encodedContent|dd-mm-yyyy hh:mm|id

Example:
trang.ptt@7-eleven.vn|Da%20kiem%20tra%20%7C%20can%20bo%20sung|17-04-2026 15:42|NP0112255C4F-17042026154233123

Encoding:
- Backend encodes only content with encodeURIComponent.
- This protects pipe characters, percent signs, and newlines.
- Parser treats invalid/nonconforming lines as legacy read-only blocks.

ID rule:
- Generated backend only.
- Format: NewProductID-ddmmyyyyhhmmssSSS.
- Collision guard increments timestamp milliseconds under lock.

Chat Backend Public APIs
Defined in Chat_Service.js:
- getProductChatHistory(payload)
- addProductChatMessage(payload)
- editProductChatMessage(payload)
- deleteProductChatMessage(payload)

Every mutation:
- Gets ScriptLock.
- Re-resolves latest storage target.
- Re-reads latest cell value.
- Parses latest transcript.
- Applies mutation.
- Writes serialized transcript back.
- Appends cell note history to the physical target cell.
- For EDIT/DELETE, writes Log_Chat.

Chat Storage Source Rules
This was changed after earlier database-only logic. Do not revert to database-only.

Storage resolver:
- _resolveProductChatStorageTarget_(payload, access, productId) in Chat_Service.js.

Rule:
- If user is in Category view and product effectiveSheet is Category, read/write Category[SSV Note Product].
- Otherwise read/write Database[SSV Note Product].

Cases:
- Category current item: UI in Edit Workspace, storage Category.
- Category viewing item that has moved to Manager/Director/etc via Process: UI in Process, storage Database.
- Manager current item: UI in Edit Workspace, storage Database.
- Manager item moved to Director/etc: UI in Process, storage Database.
- Director/Present List/Final Information/Setup Product ID: storage Database, never view/formula sheet cells.

Note History
Existing cell-note history behavior must be preserved.

When chat changes:
- If storage target is Category, append note to Category[SSV Note Product].
- If storage target is Database, append note to Database[SSV Note Product].

Note format:
yyyy-mm-dd hh:mm:ss | Sheet: <business sheet context>
Old: ...
New: ...
By: ...

Important:
- Sheet in the note is business context from the web view, such as Category or Manager.
- It is not blindly the physical storage sheet.
- For example Category user replying via Process to a Manager item stores in Database but note says Sheet: Category.

Log_Chat
EDIT and DELETE write to Log_Chat. CREATE/add does not require separate Log_Chat if the message append succeeded.

Current Process Selection Behavior
Process Tracking queue-card selection has been separated from search/open-by-query behavior.

Implemented behavior:
- _buildProcessListItemHtml calls App.selectProcessQueueItem(key).
- selectProcessQueueItem delegates to selectProcessProduct with source=queue-select and focusSearch=false.
- Clicking a queue card keeps the modal open, keeps the current queue, keeps the search input unchanged, sets S.processActiveKey, highlights the selected card, loads the right detail panel, and then loads chat for the selected product.
- Queue selection does not call searchProcess(), searchProcessQueue(), or _requestProcessingProducts().
- Search only runs when the user clicks the Search button or presses Enter in the search input.
- Exact code lookup remains a separate fallback via lookupProcessExactCode().
- Detail and chat requests use sequence/key guards so older responses cannot overwrite a newer selection.
- When a search/filter is run intentionally, the current selection is kept if it still exists in the returned queue; otherwise selection/detail/chat are cleared deliberately.

- The Process Tracking and chat text touched in the latest pass has been localized to Vietnamese while keeping sheet/header names such as Product Group and New Product ID where they are real data labels.
